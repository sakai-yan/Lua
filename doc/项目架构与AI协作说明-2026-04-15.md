# 项目架构与AI协作说明

日期：2026-04-15

## 1. 文档目标

本文档面向 AI 大模型、自动化代理与新接手开发者，目标不是介绍 Warcraft 3 或 Lua 基础，而是快速说明本仓库的：

- 结构分层
- 运行入口
- 生成链路
- 模块职责
- 工程约束
- 编辑边界

如果只想快速建立正确心智模型，可以先读第 2、3、4、8、9 节。

## 2. 项目一句话定义

这是一个以 Warcraft 3 Lua 地图开发为核心的多层工程仓库，主体玩法代码位于 `Code/`，地图展开产物位于 `War3/`，导表/打包/修图等辅助工具位于 `.tools/`，外部参考基线位于 `.canon/` 与 `Reference (not included in the project)/`，并通过 `.planning/` 维护跨任务的工程记忆与协作上下文。

它不是“只有一份地图脚本”的轻量仓库，而是同时包含：

- 运行时代码
- 地图展开产物
- 对象数据导出规则
- 编辑器/toolchain 参考基线
- 项目级规划与文档治理资产

## 4. 启动链路与执行模型

### 4.1 双入口结构

项目存在两个应同时理解的入口层：

| 文件 | 角色 |
| --- | --- |
| `Code/Init.j` | 地图侧导入桥，负责在 JASS/Lua 运行环境中引入 Lua 文件并调用 `MHLua_RunScript("Lua\\Main.lua")` |
| `Code/Main.lua` | Lua 主启动文件，按层加载模块并最终调用 `Game.execute()` |

其中：

- `Init.j` 更接近地图脚本桥接入口
- `Main.lua` 更接近工程级运行时装配入口

### 4.2 `Main.lua` 的显式装载面

`Code/Main.lua` 当前显式按如下顺序装载：

1. `Runtime`
2. `Debug`
3. `Game`
4. `Lib.API.*`
5. `Lib.Base.*`
6. `FrameWork.GameSetting.*`
7. `FrameWork.Manager.*`
8. `Core.Entity.*`
9. `Core.UI.*`
10. `Logic.Define.Attribute`
11. `Logic.Process.Unit.*`
12. `Logic.Process.Ability.*`
13. `Logic.Process.Item.*`
14. `Logic.Process.Damage`
15. `Logic.Process.Trade`
16. `Game.execute()`

这是一条“显式启动骨架”，不是完整依赖图。部分模块会通过内部 `require` 被传递装载，因此判断某模块是否生效时，不应只看它是否直接出现在 `Main.lua`。

### 4.3 `Game.execute()` 的作用

`Code/Game.lua` 维护一个初始化回调队列：

- 各模块可通过 `Game.hookInit(callback)` 注册初始化逻辑
- 所有显式模块加载完成后，由 `Game.execute()` 统一执行

这意味着项目是典型的“先装载定义，再统一初始化”的装配式启动模型，而不是每个模块 `require` 后立刻完成全部世界状态注册。

### 4.4 调试模式默认开启

`Code/Debug.lua` 当前会：

- 设置全局 `__DEBUG__ = true`
- 注册测试入口
- 创建聊天触发器用于调试

这对 AI 非常重要：当前默认启动面不是纯生产态，而是带明显调试副作用的开发态骨架。修改启动链或性能热路径时，必须考虑 `__DEBUG__` 分支和测试触发逻辑。

## 5. 运行时代码分层

### 5.1 `Lib/`：基础能力层

`Lib/` 提供通用语言级能力，定位是项目的“标准库增强层”。

#### `Lib/API`

- `Jass.lua`：把 `jass.common`、`jass.japi`、`jass.code` 统一封装到懒加载表 `Jass`
- `Constant.lua`：常量层
- `Tool.lua`：运行时工具函数

这是 Lua 与 Warcraft/JASS 运行时的主要桥接面。

#### `Lib/Base`

主要承担容器、类型系统与 OOP 设施：

- `Class.lua`：核心类系统，支持继承、getter/setter、元类、静态成员、弱引用模式
- `DataType.lua`：运行时 datatype 标注
- `Array.lua`、`Queue.lua`、`Stack.lua`、`LinkedList.lua`、`Set.lua`
- `Vector3.lua`、`HandleId.lua`
- `table.lua`、`string.lua`、`math.lua`

其中 `Class.lua` 是项目骨架级模块。大量实体类、事件类、UI 类都建立在它之上。

### 5.2 `FrameWork/`：框架约束层

`FrameWork/` 不是业务逻辑，而是运行时约定与管理器层。

#### `FrameWork/GameSetting`

负责定义项目级游戏语义与公共配置抽象，例如：

- 属性系统
- 目标类型
- 施法类型
- 地形工具
- 位置附着语义

其中 `Attribute.lua` 是关键基础设施。它不仅定义属性，还定义属性变化事件、奖励传播、真实属性名与总属性名之间的关系。

#### `FrameWork/Manager`

负责通用运行时管理器：

- `Event.lua`：事件类型系统、事件池化、按 role/object 分发
- `Timer.lua`：高性能中心定时器
- `Async.lua`：轻量异步/tick 调度层
- `Point.lua`：坐标对象
- `IDGenerator.lua`
- `State.lua`：有限状态机与 actor -> fsm 映射

这里的设计风格非常明确：

- 追求热路径短
- 尽量在初始化/编译期生成专用函数
- 大量使用对象池、弱表、按角色分发

### 5.3 `Core/`：对象包装与基础运行时

`Core/` 是“游戏对象抽象层”。

#### `Core/Entity`

主要对象族：

- `Unit`
- `Item`
- `Effect`
- `Buff`
- `Ability`
- `Player`
- `Region`
- `Missile`

这层的职责不是玩法，而是：

- 对原生 handle 做 Lua 包装
- 统一属性访问接口
- 管理 `handle -> 对象实例` 映射
- 定义创建/删除事件

以 `Unit.lua` 为例，它同时承担：

- `Unit.unitType(...)` 的类型定义
- `Unit.create(...)` 的实例创建
- `Unit.getByHandle(...)` 的反查
- 坐标/生命/法力/模型/所有者等属性访问器

它是“实体模型 + 句柄包装 + 生命周期事件源”的组合层。

#### `Core/UI`

这是仓库里最成熟、最成体系的子系统之一。根据现有分析文档，其主分层可概括为：

`Component -> Theme -> Style -> Frame`

角色大致如下：

- `Frame.lua`：所有 UI 对象的抽象基类
- `Panel.lua`、`Text.lua`、`Button.lua` 等：具体控件层
- `Theme.lua`：主题、token、selector 规则
- `Style.lua`：布局编译器
- `Component.lua`：声明式工厂入口
- `Tween.lua`：补间动画
- `Watcher.lua`：属性监听

AI 修改 UI 代码时，应优先沿这条层次理解，而不是把所有 UI 文件视为平级工具函数集合。

### 5.4 `Logic/`：业务流程与高层系统

`Logic/` 是项目的玩法流程层，分成 `Define`、`Process`、`System`。

#### `Logic/Define`

- 当前主要是属性层面的高阶定义，如 `Logic/Define/Attribute.lua`

#### `Logic/Process`

这是默认运行面的核心业务流程层，当前包含：

- 单位创建/删除/经验
- 技能添加/移除/施法
- 物品创建/删除/获得/失去/使用
- 伤害
- 交易

此外还存在若干更独立的子系统型流程模块：

- 运动：`Motion/*`
- 碰撞：`Collision.lua`
- 投射物创建：`Missile/Create.lua`
- 物品存储：`Item/Storage.lua`

这些模块当前更像“按需引入的可复用能力”，不一定在默认主路径中显式接入。

#### `Logic/System`

当前更像实验成熟中的独立上层库：

- `Shop.lua`
- `Storage.lua`
- `Group.lua`

其中：

- `Logic.Process.Item.Storage.lua` 偏事件化、面向当前项目 item 生态
- `Logic.System.Storage.lua` 更像通用容器库
- `Logic.System.Shop.lua` 是纯上层商店运行时，不直接等同于原生交易事件

## 6. 当前已成型的核心运行模式

### 6.1 事件驱动是主干，不是附属能力

本项目不是“函数直调为主，事件为辅”。从 `Event.lua`、`Unit_Create_Event`、`属性变化`、`伤害`、`物品存储变更` 等设计看，事件分发本身就是核心编排机制。

典型模式是：

1. 定义 event type
2. 事件实例池化复用
3. 按 role/object/type 分发
4. 执行任意监听
5. 回收事件实例

这意味着 AI 在扩展玩法时，优先考虑：

- 是否应挂到已有事件链
- 是否应定义新的 event type
- 是否应复用已有 role 语义

而不是优先写旁路函数。

### 6.2 “类型定义 + 实例包装”是实体层常态

例如 `Unit`：

- `unit type` 是配置/模板
- `unit instance` 是运行时对象
- `handle` 是底层引擎对象

三者不可混淆。很多 API 接口显式区分：

- `unit`
- `unittype`
- `item`
- `itemtype`
- `player`

AI 修改相关代码时，必须先确认参数语义属于“类型”还是“实例”。

### 6.3 高性能热路径偏好非常明确

从 `Class.lua`、`Timer.lua`、`Async.lua`、`Storage.lua`、运动库设计与项目 playbook 可以归纳出稳定倾向：

- 预处理前置
- 热路径避免重复校验
- 偏好 O(1) 核心变更
- 只抽真正高价值 helper
- 直接数据流优先于装饰性抽象

这不是局部风格，而是项目级约束。

## 7. 配置、导表与生成物链路

### 7.1 `Code/Config/` 的真实定位

`Code/Config/` 不是普通业务模块，更接近“对象数据描述层 + 导出规则层”。当前包含：

- `Mapper.lua`
- `UnitTypeInit.lua`
- `UnitType/*.lua`
- `ItemType/*.lua`
- `Skill/*.lua`
- `Template.md`

其中：

- `Mapper.lua` 定义中文字段到物编字段的映射、值映射与默认值策略
- `UnitType/*.lua` 是单位模板定义源
- `UnitTypeInit.lua` 带有导表/生成痕迹，承担聚合与初始化角色

### 7.2 Packer 的职责边界

根据 `.tools/Packer` 文档，Packer 负责：

- 把项目源码输出到 War3 运行目录
- 补全 `UnitType` 数据
- 生成 `map/Config/UnitTypeInit.lua`
- 生成 `table/unit.ini`

字段优先级为：

`default < Excel < Lua`

因此，如果任务涉及单位物编/导表，不应直接手改 `War3/table/unit.ini` 作为长期方案，而应优先回到：

1. `Code/Config/Mapper.lua`
2. `Code/Config/UnitType/*.lua`
3. Excel 源
4. Packer 规则

### 7.3 `War3/` 的定位

`War3/` 是地图展开产物层，包含：

- `map/`：地图展开后的对象与脚本文件
- `resource/`：资源
- `table/`：ini 类导表产物
- `.w3x`：打包后的地图文件

这里更像输出目录或编辑器可消费目录，而不是最优先的手工维护源。

## 8. 非代码资产与辅助工程边界

### 8.1 `.tools/`

这是项目本地工具集合，不应与主玩法代码混淆。

当前重点工具包括：

- `.tools/Packer`：打包/导表/同步工具
- `.tools/MapRepair`：YDWE 可开图修复工具
- `.tools/lua53`：本地 Lua 5.3 工具链
- `.tools/luamake`
- `.tools/YDWE`

这些目录往往是独立工程或独立发布物。只有当任务明确涉及工具链、打包、修图、验证时，才应把它们当主编辑面。

### 8.2 `.canon/`

`.canon/` 是 canonical reference，当前包含：

- `w3x2lni`
- `YDWE`

它的角色是：

- 对照
- 考证
- 回溯
- 参考实现

不是当前业务代码的默认编辑面。除非任务明确要求维护基线参考，否则应视为只读。

### 8.3 `Reference (not included in the project)/`

该目录名已经明确说明其语义：

- 用于参考
- 不视为当前项目源码
- 不应把其中变更当作当前项目已接入事实

AI 阅读它可以获得设计灵感或对照证据，但不能默认其 API、约束、目录结构已经在本项目内生效。

### 8.4 `.planning/`

这是规划与项目记忆层，不是运行时代码。

其中：

- `.planning/PROJECT_PLAYBOOK.md`：项目级共享规则
- `.planning/workspaces/*`：具体任务工位

对 AI 而言，这一层的价值主要是：

- 提供稳定风格基线
- 记录已确认的项目约束
- 减少重复调查

## 9. 默认主路径与候选系统的边界

AI 理解本仓库时，应把模块分成三类：

### 9.1 默认主路径

指 `Main.lua` 显式装载并通过 `Game.execute()` 纳入当前运行骨架的部分，典型包括：

- 基础 API 与容器库
- 游戏设置与管理器
- 实体包装层
- UI 基础框架
- 单位/技能/物品/伤害/交易流程

### 9.2 传递依赖

指不一定在 `Main.lua` 直接列出，但会被已装载模块内部 `require` 的依赖，例如：

- `Core.UI.Theme`
- 部分地形/状态/底层辅助模块

这类模块通常是主路径的一部分，但不是“显式入口面”。

### 9.3 候选系统或按需系统

当前仓库中存在但不应默认视为“已纳入当前主循环”的模块，例如：

- `Logic.Process.Motion.*`
- `Logic.Process.Collision`
- `Logic.Process.Missile.Create`
- `Logic.Process.Item.Storage`
- `Logic.System.Shop`
- `Logic.System.Storage`
- `View.Buff`
- `Code/Script/`

它们的状态更接近：

- 能力库
- 预备系统
- 草稿/扩展点
- 独立任务产物

AI 在接需求时，不能因为目录存在就假设这些系统已经全链路接通。

## 10. 项目级编码风格与实现倾向

根据现有 `doc/项目代码风格总结-2026-04-11.md` 与 `.planning/PROJECT_PLAYBOOK.md`，当前项目默认基线可以归纳为：

- 倾向扁平字段，避免为了结构美观额外套层
- 初始化阶段多做归一化、校验和派生值缓存
- 热路径信任已建立的内部契约
- 公共入口保持薄，`prepare/resolve` 与 `apply/mutate` 分层明确
- 仅抽取真正共享、真正值钱的 helper
- 优先复用项目已有能力，不重复造薄工具
- 公开注释强调契约、复杂度、成本模型、边界与示例

对 AI 来说，这意味着：

- 不要轻易引入“看起来更抽象”的中间层
- 不要把简单逻辑拆成很多低价值包装
- 不要把一次任务中出现的行为改动误当成全局风格

## 11. AI 修改项目时的推荐策略

### 11.1 最安全的主编辑面

优先顺序建议如下：

1. `Code/`
2. `doc/`
3. `.planning/` 中当前任务工作区

### 11.2 需要谨慎或默认避免的编辑面

- `.canon/`
- `Reference (not included in the project)/`
- `War3/.w3x`
- `War3/map/*` 与 `War3/table/*` 中明显由导出生成的产物
- `.tools/` 中与当前任务无关的独立工具工程

### 11.3 处理不同类型需求的正确落点

| 需求类型 | 首选落点 |
| --- | --- |
| 玩法流程修改 | `Logic/Process` 或 `Logic/System` |
| 引擎对象包装 | `Core/Entity` |
| 属性与数值传播 | `FrameWork/GameSetting/Attribute.lua` |
| UI 组件/主题/布局 | `Core/UI` |
| 通用容器/类工具 | `Lib/Base` |
| 物编字段/对象模板/导表规则 | `Code/Config` + `.tools/Packer` |
| 地图修复/WTG 相关 | `.tools/MapRepair` |
| 上游工具或编辑器参考比对 | `.canon` / `w3x2lni-source` |

## 12. AI 需要特别警惕的误判点

### 12.1 不要把产物目录当源码真源

`War3/` 很重要，但更多是地图展开产物层。除非任务明确是“修改导出结果”或“直接修地图产物”，否则不应把它当第一编辑面。

### 12.2 不要把参考目录当项目事实

`.canon/`、`Reference (not included in the project)/`、`w3x2lni-source/` 都能提供证据，但不等于当前项目已经采用其全部契约。

### 12.3 不要假设所有现存模块都已接入主流程

目录存在、文件完整、文档完善，不代表已经进入默认启动链。必须区分：

- 已显式装载
- 传递装载
- 按需装载
- 仅设计/仅实验

### 12.4 不要忽略全局调试态

`__DEBUG__ = true` 会影响：

- 断言
- 测试入口
- 性能路径
- 调试输出

任何“优化”“清理副作用”“变更初始化顺序”类工作，都要先评估调试态行为。

### 12.5 不要混淆中文语义字段与底层物编字段

项目在配置层大量使用中文语义字段，再由 `Mapper.lua` 映射到物编字段。这是有意设计，不应粗暴替换为底层字段名后到处直写。

## 13. 面向 AI 的快速结论

如果一个模型只记住下面这些结论，通常就不会走偏：

1. 这个仓库的真正主代码在 `Code/`，不是 `War3/`、`.canon/` 或 `Reference`。
2. 启动骨架是 `Init.j -> Main.lua -> Game.execute()`。
3. 事件系统、属性系统、实体包装层是运行时主干。
4. UI 子系统已经形成 `Component -> Theme -> Style -> Frame` 的稳定分层。
5. `Config` 是导表/物编桥接层，不是普通玩法逻辑层。
6. `.tools/Packer` 和 `.tools/MapRepair` 是并行维护的工具工程，不要和主玩法代码混编。
7. 项目风格偏向直接、扁平、预处理前置、热路径短。
8. 默认不要编辑 `.canon/`、`Reference (not included in the project)/`、打包产物 `.w3x`。
9. 当前仓库里“存在的模块”不等于“默认主路径已接入的系统”。
10. 做改动前先判断任务属于：运行时代码、配置导表、地图产物、工具工程、参考基线，避免落错层。
