# 项目代码风格总结

日期：2026-04-11

## 背景

这份文档不是重新发明一套“理想风格”，而是把项目里已经明确表达过的一套风格正式收束起来。

它主要来自两层历史证据：

1. 2026-04-10 已经写过一次的 [Bezier3代码风格总结](./Bezier3代码风格总结-2026-04-10.md)。
2. `Charge`、`Surround`、`Bezier3`、`Vertical`、旧版 `Down` 这几类运动库在整理、改写、补全过程中反复出现的共同结构。

其中要特别注意：

- `Charge.lua` 和旧版 `Down.j` 更像是这套风格的早期脉络。
- `Bezier3.lua` 与 `Vertical.lua` 更接近这套风格已经被明确提纯后的样子。
- 如果旧文件写法与后来的明确总结冲突，应优先以后来的明确总结为准，而不是机械照抄更旧的实现细节。

## 一句话版本

整个项目今后在一般情况下，默认优先采用这样一种写法：

“代码尽量写得扁平、直接、拥有者明确、初始化期多做准备、热路径短直、只抽真正共享的核心逻辑、API 收敛、注释围绕契约/代价/示例，而不要为了抽象完整、兼容好看、层次漂亮，再把代码写重。”

## 整个项目默认参照的代码风格

### 1. 数据和接口尽量扁平，不为形式美观额外套层

默认优先：

- 使用扁平标量字段。
- 让字段语义一眼可见。
- 让调用方直接知道自己在传什么。

一般不要为了“看起来整齐”额外做这些事：

- 把本来就是几个标量的输入临时包成 table。
- 再套一层只为分组而存在的对象。
- 在没有真实调用需求时保留兼容层。

例外情况：

- 外部调用契约本来就是嵌套结构。
- 某个对象本身就有天然整体语义。
- 兼容层确实有真实调用方需要。

## 2. 运行态和派生态直接挂在拥有者表上

项目更偏好的方式是：

- `config` / `motion` / `self` / 归属明确的实例表，直接承载运行态。
- 派生值、缓存值、阶段性字段直接写在拥有者身上。

不优先的方式是：

- 为了“结构完整”再引入一层中间状态对象。
- 明明只有本模块自己持有，却再拆一层包装。

核心判断标准不是“抽得够不够优雅”，而是：

- 所有权是否明确。
- 阅读路径是否更短。
- 热路径是否更干净。

## 3. 初始化阶段多做准备，执行阶段少做判断

这是当前项目里最稳定的一条风格线。

默认倾向：

- 在入口或 prepare 阶段完成必要校验。
- 在 setup 阶段完成默认值归一、方向归一、派生值缓存、系数缓存。
- 让 update / tick / 核心执行路径尽量只做真正每次都必须做的数值推进和状态落位。

换句话说：

- 能在初始化时做完的事，就不要拖到热路径里每次再做。
- 能提前统一语义，就不要在每次执行时反复分支。

## 4. helper 只抽真正共享、真正值钱的部分

这套风格不是“不要 helper”，而是“不要低价值 helper”。

适合抽 helper 的情况：

- 真正被多个分支共享。
- 抽出来以后确实缩短了核心阅读路径。
- 抽出来以后不会把逻辑切碎得更难追。

一般不建议抽 helper 的情况：

- 只为“层次完整”多拆一层。
- 只服务于一两个局部点，却把阅读路径打断。
- 把本来应该就地展开的变体初始化硬拆走。

因此整个项目更偏向：

- “共享核心逻辑可以抽”
- “变体细节尽量靠近变体本身”

## 5. 公共入口要薄，主逻辑分层要直接

项目里更偏好的结构通常是：

1. 公共入口函数做最必要的参数判断和分发。
2. `prepare / resolve` 负责规范化输入、补默认值、缓存派生值。
3. `update / apply / mutate` 负责真正推进状态。

也就是说：

- 公共入口不宜又厚又杂。
- 但也不宜为了“薄”而把逻辑拆成很多低价值中转。

最理想的状态是：

- 入口薄。
- 准备层清楚。
- 热路径一眼能扫明白。

## 6. 一旦内部契约成立，热路径就应信任契约

这也是从运动库里提炼出来、但适用于整个项目的风格。

项目偏好不是“完全不校验”，而是：

- 入口保留必要校验。
- 契约一旦建立，后续路径就按契约直接走。

不鼓励的写法：

- 每一层都重复防御。
- 热路径里反复做入口级别的兜底判断。
- 把内部模块写成“超强兼容适配器”。

## 7. 优先复用项目里已经存在的稳定能力

如果项目环境已经有成熟能力，优先复用，而不是为了文件“自洽”再本地重写一层很薄的小工具。

优先复用的对象包括：

- 数学基础能力
- 统一入口/生命周期
- 已存在的实体 setter
- 已存在的共享结构和工具

这条规则的本质不是“少写代码”，而是：

- 避免重复定义同一能力。
- 避免制造多份相似但不完全一致的小轮子。

## 8. API 要收敛，不为了好看保留别名和兼容壳

默认原则：

- 先收敛主名。
- 先明确唯一推荐入口。
- 没有真实调用需求时，不主动保留别名、旧名、兼容层。

如果确实有旧调用方、迁移成本、外部约束，那是另一回事。

但在“一般情况”下，整个项目后续设计应默认：

- 不为了“看起来友好”增加维护面。

## 9. 模块扩展时按运行模型/核心抽象分组，而不是按名字堆分支

这条规则最早从运动库里变得清晰，但它不只适用于运动库。

当一个模块开始从单点功能扩成一个小家族时，更推荐：

- 先按真正的运行模型分组。
- 先按共享核心抽象分层。
- 薄包装只在“完全复用已有实现”时才成立。

不推荐：

- 每加一个语义名字就复制一套新分支。
- 为了名字齐全让实现越来越碎。

## 10. 代价模型要明确写出来

如果一个接口：

- 是 O(1) 的核心路径，那就保持紧凑直接。
- 是 O(n) / O(capacity) / 会分配新表 / 会扫描，那就应该在注释里明确说明。

项目更偏好的注释风格不是“功能说明越多越好”，而是：

- 调用契约清楚。
- 代价模型清楚。
- 边界和 caveat 清楚。
- 能给出可直接照写的示例。

## 11. 注释重点放在“契约、代价、原因、示例”

当前项目更偏好的注释策略是：

- 公共接口上方写清调用契约。
- 写清字段语义、默认值、结束行为、复杂度或代价。
- 给出简洁可复制的使用示例。
- 内部注释只写真正值得写的设计点。

不建议：

- 对每个显然语句都做逐行翻译式注释。
- 用大量注释去掩盖层次过重的问题。

## 12. 优先做叶子模块内的解法，不轻易扩大到共享框架改造

这更像项目设计边界风格，但也值得一起记住。

默认优先：

- 先在当前叶子模块内解决。
- 框架只有在真的构成阻塞时才扩大改动面。

这样做的好处是：

- 风险小。
- 边界清楚。
- 后续回看更容易知道变化归属。

## 哪些不应被机械当成“全项目硬规则”

下面这些内容不能因为在某次风格优化里出现，就一律上升成机械模板：

1. 某个具体行为契约是否保留历史兼容。
2. 某个模块是否检查 `target.alive == false`。
3. 结束条件是直接 `return`，还是先落到 `need_to_end` 再返回。
4. 某个默认值到底是放宽还是收紧。
5. 某个模块是否需要多一层兼容对象来适配真实外部协议。

也就是说：

- 可以学“结构上的更直接”。
- 不能把每一次行为删改都自动当成全项目模板。

## 冲突时如何判断

以后如果出现“旧代码是这样写的，但新总结不是这样说”的情况，默认按下面顺序判断：

1. 先看是否已有明确的后续总结或 playbook 规则。
2. 再看这个差异到底是风格，还是行为契约。
3. 再看是否存在真实调用方、兼容压力、外部协议约束。
4. 如果都没有，再按当前这份项目风格基线处理。

## 后续使用约定

从 2026-04-11 开始，这份文档和 `.planning/PROJECT_PLAYBOOK.md` 中同步后的规则，视为本项目后续代码设计在一般情况下的默认参照风格。

如果某次任务明确偏离这套风格，最好满足至少一个条件：

- 有真实调用契约需要。
- 有明确性能/兼容/框架约束需要。
- 偏离原因能在注释、任务文档或评审说明中说清楚。
