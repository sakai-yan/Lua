# Bezier3代码风格总结

日期：2026-04-10

## 背景

本文基于当前 `Code/Logic/Process/Motion/Bezier3.lua` 的用户优化版本整理。

对比对象主要有两类：

1. 兄弟运动库 `Charge.lua` / `Surround.lua`
2. 本任务中更早一版的 Bezier3 设计思路

前一版设计思路的特点更偏向：

- 保留更多兼容层
- 用更多小 helper 做输入修正与系数缓存
- 更强调“安全兜底”和接口整齐

而用户当前版本表达出的要求更明确：

- 更扁平
- 更直接
- 更少包装
- 更贴近 `Charge.lua`
- 更愿意信任调用契约

这份文档的目标，不是评价哪一版“绝对更好”，而是沉淀出当前项目里你真正要求的代码风格。

## 结论先行

从这次 Bezier3 的改写可以总结出，当前项目中你偏好的运动库写法大致是：

1. 输入配置优先使用扁平标量字段，不为语义分组额外创建坐标 table。
2. 运行态直接挂在 `config` / `motion` 上，不再引入额外状态对象。
3. 真正共用的核心逻辑才提 helper；变体逻辑宁可就地展开，也不要为了“抽象完整”继续拆层。
4. 热路径尽量信任初始化后的契约，减少防御性分支和重复校验。
5. 优先复用项目里已经存在的基础能力，不重复发明小工具。
6. API 面尽量收敛，避免为了“兼容感”保留多余别名和过渡层。
7. 注释应该服务于调用者理解和维护者扫描，不追求“面面俱到”的包装式说明。

## 这次风格变化里，最值得记住的点

### 1. 接口层要扁平，不做坐标 table 包装

当前版本把：

- `start_point`
- `end_point`
- `control_one`
- `control_two`

改成了：

- `start_x/start_y/start_z`
- `end_x/end_y/end_z`
- `control_one_x/control_one_y/control_one_z`
- `control_two_x/control_two_y/control_two_z`

这说明当前风格更重视：

- 调用成本可见
- 字段语义直接
- 少一层构造/拆包
- 和 `Charge.lua` 那种标量配置风格统一

也就是说，在这个项目里，“三维点”更适合作为已有对象的自然形态，例如 `target.x/y/z`，而不是为了参数整洁临时组一个 table。

## 2. 只保留真正高价值的共享 helper

当前版本保留了 `moveBezier()`，因为它同时被固定终点和锁定目标两条路径复用，而且它就是核心热路径。

但当前版本去掉或弱化了以下类型的 helper 倾向：

- 单纯为了“输入规整”而存在的小函数
- 单纯为了“抽象完整”而拆出去的缓存函数
- 只有一两处使用、但又把阅读路径切碎的中间 helper

这说明你的风格更偏向：

- 共用核心逻辑可以抽
- 变体初始化逻辑尽量靠近各自的 prepare 函数
- 不要把一件本来连续的事情拆成过多跳转

简单说，就是“共享的核心抽出来，分支的细节留在分支里”。

## 3. 运行时逻辑追求直线化

当前版本的 update 路径很接近你想要的风格：

- 取 `motion`
- 推进 `t`
- 计算位置
- 落位
- 返回结束状态

这种写法的特点是：

- 扫一遍就能看懂
- 热路径没有多余绕行
- 真正影响性能和行为的代码聚在一起

对应地，不太符合当前偏好的写法是：

- 先过很多层包装函数
- 热路径里掺杂过多输入修正
- 每帧都做“本可在初始化完成”的判断

## 4. 初始化阶段多做事，tick 阶段少做事

当前版本仍然保留了初始化时的系数缓存，但把策略进一步往“就地准备好”推进：

- 公共 prepare 只处理共享默认值和共享运行态
- 固定终点版自己写完整系数
- 锁定终点版自己写 base 系数

这背后体现的是一个很明确的取向：

- 初始化逻辑可以适度展开
- tick 热路径必须尽量短
- 不要为了初始化函数好看，牺牲 tick 的直接性和代码扫描效率

## 5. 更信任调用契约，减少防御性包装

和前一版相比，当前版本明显更少使用：

- 层层 `type(...)` 修正
- 兜底读点函数
- 输入兼容层
- 重复的运行时保护

这说明你的偏好不是“完全不校验”，而是：

- 入口保留必要校验
- 共享契约一旦确立，后面就按契约直接跑
- 不要把 leaf module 写成“超强兼容适配器”

这种风格尤其适合当前这类内部运动库。

## 6. 优先复用已有能力，不本地再造小轮子

当前版本使用了 `math.clamp`，而不是继续保留本地 `clamp01`。

这类变化说明你的风格偏好是：

- 如果项目环境里已经有稳定基础能力，优先复用
- 不要为了“文件自洽”重复写一份非常薄的小工具

这个原则同样适用于别的库：

- 数学工具
- 基础常量
- 通用 runtime 入口
- 共享 move/set/get 机制

## 7. API 面要收敛，不滥留别名

当前版本去掉了额外别名暴露，保留主入口即可。

这说明你的 API 风格更偏向：

- 先收窄
- 先明确唯一主名
- 不为“看起来友好”增加额外维护面

换句话说，如果没有明确兼容需求，就不要默认同时保留多套调用名。

## 8. 注释要围绕“调用理解”和“设计取向”

当前版本的注释有一个明显特点：

- 公共函数上方仍然保留配置说明和用法
- 内部注释只写关键设计点
- 不把每一个小函数都写成大段“说明书”

这说明当前偏好的注释策略是：

- 对外接口多解释一点
- 对内实现只解释读者不容易一眼看懂的部分
- 注释服务于扫描和维护，不追求注释数量本身

## 与前一版设计相比，哪些变化可以上升为“风格要求”

以下变化可以基本视为当前明确的风格要求：

1. 不要为坐标参数额外创建 point table。
2. 不要保留兼容 point table 的适配层。
3. 不要为了整齐继续拆太多 helper。
4. 能用项目已有能力就不要本地重写薄工具。
5. 共享逻辑只抽真正共享且高价值的部分。
6. 入口字段尽量与兄弟库保持同类风格的扁平语义。
7. update 路径保持短、直、可扫。

## 哪些属于“行为选择”，不应机械当成风格本身

有些变化虽然和当前版本一起出现，但它们更像“行为契约选择”，不应简单上升为普遍风格规则：

1. 是否保留历史别名。
2. 是否检查 `target.alive == false`。
3. 结束条件写成直接 `return`，还是写成 `local need_to_end` 再返回。
4. 某些入口默认值是否允许继续放宽。

这些点以后仍应根据：

- 是否已有调用方
- 是否和兄弟库行为对齐
- 是否影响脚本兼容

来单独判断。

也就是说，风格上可以学“更直接”，但不能把每一次行为删改都自动当成风格模板复制。

## 可复用的落地模板

以后写类似运动库时，可以优先按下面这套顺序思考：

1. 公共入口字段是否能全部扁平化。
2. 是否真的需要兼容旧字段；没有明确调用方就不要默认兼容。
3. 是否只有一个真正共享热路径 helper。
4. 公共 prepare 是否只保留“真正共享”的默认值和运行态准备。
5. 各变体的缓存/初始化是否应该就地展开。
6. tick 里是否还残留本该在初始化完成的逻辑。
7. 注释是否把重点放在“接口怎么用”和“设计为什么这么做”。

## 一句话版本

你这次通过 Bezier3 表达出来的代码风格要求，可以概括成一句话：

“运动库优先写成扁平、直接、信任契约、热路径短直、只抽真正共享核心逻辑的样子；不要为了抽象完整和兼容好看，重新把代码写重。”
