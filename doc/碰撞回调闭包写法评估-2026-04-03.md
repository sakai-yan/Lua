# 碰撞回调闭包写法评估 - 2026-04-03

## 1. 评估目标

本评估聚焦当前碰撞流程中的这类写法：

- 文件：[Code/Logic/Process/Collision.lua](../Code/Logic/Process/Collision.lua)
- 关键位置：
  - `execute(...)` 内联回调：`50-59` 行
  - 预定义但当前未使用的 `collideUnit(...)`：`19-28` 行
  - 预定义但当前未使用的 `collideMissile(...)`：`31-40` 行

目标不是泛泛讨论 “Lua 闭包好不好”，而是回答：

1. 你当前这份写法的真实成本模型是什么。
2. 它和可替代写法相比，CPU、内存、GC 差多少。
3. 在本项目里，这个点是否已经值得为了性能改造。
4. 如果改，应该往哪一类方案改，风险是什么。

## 2. 先说结论

### 2.1 总结判断

- 你当前的写法**不是灾难级问题**。
- 但它确实会在 `execute(...)` 每次调用时**额外分配 1 个闭包**。
- 这个闭包不是“每个被枚举单位分配一次”，而是“每次碰撞枚举调用分配一次”。
- 在**命中数量较多**时，回调主体逻辑会摊薄这部分成本，闭包创建不再是主导。
- 在**稀疏碰撞 / 空枚举很多**时，闭包创建成本会比较显眼。
- 如果未来这个系统同时跑大量运动器/投射物，且很多时候都“扫了一圈但没有命中”，那么当前写法会比“共享回调 + 外部上下文”更容易制造**无效分配和 GC 压力**。

### 2.2 工程建议

- 以**正确性 / 可维护性优先**来看，当前写法可以接受。
- 以**极致热路径优化**来看，当前写法不是最优。
- 如果你现在还没有通过 profiler 观察到 `Collision.execute(...)` 是前排热点，建议先**保留写法**，但把这份评估文档作为后续优化依据。
- 如果后面确认这里是热点，优先考虑：
  - `共享回调 + 上下文栈/上下文槽`
  - 不建议直接上“单一全局槽位”的最简版本，除非你能确认不会发生重入

## 3. 当前代码的真实形态

### 3.1 当前 `Collision.lua` 不是“纯预定义回调”模式

当前 [Collision.lua](../Code/Logic/Process/Collision.lua) 在 `execute(...)` 中实际使用的是：

```lua
Unit_enumInRange(x, y, collision.range, function ()
    local knock_target = Unit_getEnumUnit()
    local collision = fsm.collision
    if knock_target.alive and not Group.has(collision.reknock_unit_group, knock_target) then
        if type(collision.on_collide) == "function" then
            collision.on_collide(fsm, knock_target)
        end
        Group.remove(collision.reknock_unit_group, knock_target)
    end
end)
```

这意味着：

- 每次 `execute(fsm)` 进入 `fsm.can_collide_unit` 分支，都会新建一个匿名闭包。
- 这个闭包会捕获当前 `fsm`。
- 它不是每命中一个单位建一次，而是每次调用 `Unit_enumInRange(...)` 建一次。

### 3.2 `Unit.enumInRange(...)` 当前并不透传额外参数

文件：[Code/Core/Entity/Unit.lua](../Code/Core/Entity/Unit.lua)

```lua
enumInRange = function(x, y, radius, action)
    MHUnit_EnumInRangeEx(x, y, radius, action)
end
```

也就是说，当前接口只有 `action`，没有把 `fsm` 之类的上下文往回调里额外透传。

这也是为什么 [Collision.lua](../Code/Logic/Process/Collision.lua) 现在会自然写成“内联闭包捕获 `fsm`”，而不是直接把 `collideUnit` 这种带参函数传进去。

### 3.3 项目里已经存在“共享回调 + context pack”的先例

文件：[Code/Logic/Package/Group.lua](../Code/Logic/Package/Group.lua)

这里采用的是：

- 模块级 `param_pack`
- 共享回调 `addUnitToGroup`
- 枚举前先把上下文放进 `param_pack`

所以从项目风格来看，“共享回调 + 外部上下文”是已有模式，不是陌生方案。

### 4.3 专项基准比较了什么

为了贴近当前 `Collision.execute(...)` 的真实行为，专项基准比较了三种模式：

1. `inline_capture`
   - 与当前写法一致
   - 每次 `execute` 新建 1 个闭包，闭包捕获 `fsm`
2. `shared_slot`
   - 共享 1 个回调函数
   - `execute` 前把 `fsm` 放入模块级槽位
3. `shared_pack`
   - 共享 1 个回调函数
   - `execute` 前把 `fsm` 放入 `param_pack` 风格的表

专项基准还按每次枚举命中的数量分别测试了：

- `0`
- `1`
- `8`
- `32`

这样可以分别覆盖：

- 空枚举/稀疏碰撞
- 普通碰撞
- 密集碰撞

所有结果为 `3` 轮中位数。

## 5. 当前闭包到底捕获了什么

专项基准对“当前写法等价闭包”做了 upvalue 检查，结果如下：

| index | upvalue 名称 | 类型 | 多个闭包之间是否共享 |
| --- | --- | --- | --- |
| 1 | `Unit_getEnumUnit` | `function` | 是 |
| 2 | `fsm` | `table` | 否 |
| 3 | `Group` | `table` | 是 |
| 4 | `type_local` | `function` | 是 |

结论：

- 当前闭包确实有 `4` 个 upvalue。
- 其中**真正每次 `execute` 都不同**的是 `fsm`。
- 其余 upvalue 是模块级共享引用，不是每个闭包都新造一份完整副本。

这也是为什么当前写法的真实代价，虽然明显高于“无分配共享回调”，但又没有夸张到不可接受。

### 7.2 怎么读这组数据

#### 情况 A：`hits = 0`

这是“枚举调用发生了，但范围里没有目标”的情况。

这里当前写法最吃亏，因为：

- 还是分配了闭包
- 但回调主体几乎没机会摊薄这份开销

所以会出现：

- `inline_capture = 146 ms`
- `shared_slot = 30 ms`
- 差了 `4.87x`

- 差的是 `0.116 us/execute`

这个比例看着大，绝对时间其实不算离谱。

#### 情况 B：`hits = 1`

- `inline_capture` 比 `shared_slot` 慢 `47.5%`
- 绝对差约 `0.095 us/execute`

#### 情况 C：`hits = 8`

这更接近很多实际碰撞流程。此时：

- `inline_capture` 只比 `shared_slot` 慢 `16.3%`
- 绝对差约 `0.277 us/execute`

说明回调体逻辑已经开始盖过“建闭包”本身。

#### 情况 D：`hits = 32`

高密度碰撞下，主体逻辑和回调调度都很重，闭包仍然有代价，但不再是唯一成本源。

## 8. 数据三：GC 被延迟时的最坏情况

为了量化“当前写法会不会在高压情况下堆垃圾”，专项基准把 GC 暂停，并执行 `1000000` 次 `0-hit` 枚举。

| 场景 | loop_ms | heap_growth | Full GC | residue |
| --- | ---: | ---: | ---: | ---: |
| `inline_capture_gc_paused` | `141 ms` | `91.55 MB` | `72 ms` | `0 KB` |
| `shared_slot` | `38 ms` | `0 MB` | `0 ms` | `0 KB` |
| `shared_pack` | `35 ms` | `0 MB` | `0 ms` | `0 KB` |

结论：

- 当前写法在 GC 正常运行时，堆增长不明显。
- 但在“高频空枚举 + GC 延后”的情况下，会形成明显的**死闭包堆积**。
- `1000000` 次这样的调用，实测会留下约 `91.55 MB` 待回收垃圾，并在 full GC 时打出约 `72 ms` 的停顿。

这不是说你现在一定会遇到这个情况，而是说明：

- 当前写法的风险主要不是“单次回调慢”
- 而是“高频时会制造额外垃圾，GC 一旦滞后就会集中回收”

## 10. 共享回调为什么快，但不一定该直接改

### 10.1 它快的原因

`shared_slot/shared_pack` 更快的核心原因只有一个：

- **不分配闭包**

它们复用同一个回调函数，所以：

- 减少 CPU 分配开销
- 减少 GC 垃圾

### 10.2 它的真实风险

如果采用最简单的：

```lua
current_fsm = fsm
Unit_enumInRange(x, y, range, shared_callback)
current_fsm = false
```

那么你会得到一个新的风险：**上下文重入覆盖**。

例如：

1. `Collision.execute(A)` 设置 `current_fsm = A`
2. 枚举过程中，`on_collide` 又触发了别的枚举
3. 这个新的枚举把 `current_fsm` 改成了 `B`
4. 原来的枚举还没走完，回调上下文就错了

当前内联闭包不会有这个问题，因为它天然把 `fsm` 绑定在自己身上。

### 10.3 所以真正安全的优化方向不是“单槽位”，而是“上下文栈”

如果以后真的要优化，建议的方向是：

- 共享回调
- 模块级上下文栈，入栈/出栈，而不是单一槽位

这样可以同时兼顾：

- 不频繁分配闭包
- 避免重入覆盖

## 11. 对当前代码的具体评价

### 11.1 性能评价

- 当前写法在性能上属于：**合理，但不是最优**
- 它的主要成本来自：**每次 `execute` 额外创建 1 个捕获 `fsm` 的闭包**
- 这份成本在空枚举和稀疏碰撞中最显眼
- 在普通命中密度下，这份成本会被回调体逻辑摊薄

### 11.2 正确性评价

- 当前写法的上下文绑定最直接
- 不依赖共享状态
- 对重入更天然安全

所以从正确性角度，它反而比“最简共享槽位优化版”更稳。

### 11.3 可维护性评价

当前文件里有两个明显的维护信号：

1. `collideUnit(...)` / `collideMissile(...)` 目前未被 `execute(...)` 使用
2. `Unit.enumInRange(...)` 的注释里写了 `--@param ... any`，但当前实现并没有 varargs 透传

这会让后续阅读者误以为：

- 这里本来应该能直接传 `fsm`
- 或者当前代码只是“暂时没整理完”

建议至少做文档或注释层面的对齐。

## 12. 建议的决策顺序

### 方案 A：先不改实现，只保留当前写法

适用条件：

- 当前没有 profiler 证据表明 `Collision.execute(...)` 是热点
- 更看重正确性和简单性

建议动作：

- 保留现状
- 删除或整理未使用的 `collideUnit(...)` / `collideMissile(...)`
- 修正文档注释，说明 `enumInRange(...)` 当前不支持额外参数

### 方案 B：如果确认这里是热点，再升级为“共享回调 + 上下文栈”

适用条件：

- 已确认这里是 CPU/GC 热点
- 项目里同类枚举路径很多
- 愿意为热路径做结构性优化

建议原因：

- 比当前写法更省分配
- 比单槽位版更安全

### 方案 C：不建议直接上“共享回调 + 单全局槽位”

不建议原因：

- 最快
- 但也最容易踩重入上下文错乱

除非你能明确保证：

- 枚举期间不会递归进入同一类枚举
- `on_collide(...)` 链路不会再次触发依赖同一上下文槽的逻辑

## 13. 最终判断

对“像目前我这样的写法呢”这个问题，最终结论是：

- **能用，而且在很多实际场景下不会成为首要瓶颈**
- **但它确实不是热路径最优写法**
- **它的核心代价不是单次执行慢，而是高频下持续制造闭包垃圾**
- **如果未来这个路径被 profiler 证明确实很热，值得改；如果没有证据，当前写法完全可以先保留**

一句话概括：

- 当前写法偏“稳”和“直观”
- 共享回调方案偏“快”
- 真正值不值得切换，取决于这里是不是已经成为你项目里的前排热点
