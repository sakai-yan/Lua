# 运动器与碰撞框架设计

## 1. 文档目标

本文档用于明确当前项目中 `unit`、`effect`、`missile` 三类实体在“运动”和“碰撞”上的职责边界，并给出一套适合当前 Lua 工程逐步落地的框架设计。

本文档只做设计说明，不要求立即改动现有代码。重点是先把链路和模块边界讲清楚，让后续重构时有稳定的落点。

## 2. 当前问题的本质

从当前 `Code/Logic/Motion/Effect/Entry.lua` 的语义看，`motion` 已经同时承担了以下多种职责：

- 保存实体运动状态
- 每帧驱动位移
- 做碰撞检测
- 保存碰撞去重信息
- 调用命中事件
- 决定运动结束和销毁

这会带来几个长期问题：

- `motion` 既是“运动器”，又像“碰撞器”，还像“命中处理器”，职责过重。
- `missile` 在创建时天然就需要具备碰撞逻辑，因此它不能等“之后再挂碰撞”，而必须在出生时就进入完整运行态。
- `unit`、`effect`、`missile` 三类实体对运动和碰撞的需求不同，但当前结构容易把它们揉进同一套分支逻辑里。
- 后续一旦增加新的轨迹类型、碰撞策略、穿透规则、反弹规则，`Entry.lua` 会持续膨胀。

因此，真正需要拆开的不是实体种类本身，而是运行时能力。

## 3. 总体设计原则

### 3.1 实体负责“是什么”，系统负责“怎么跑”

- `unit`、`effect`、`missile` 是实体。
- `motion`、`collider`、`hit policy` 是挂在实体上的运行时能力。
- `motion system`、`collision system` 是每帧驱动这些能力的系统。

实体不要自己写完整流程，实体只持有必要配置和能力引用。

### 3.2 将“碰到谁”与“碰到后怎么办”分开

碰撞逻辑建议拆为两层：

- `collider`：负责检测是否发生碰撞
- `hit policy`：负责处理碰撞结果

这两个概念必须分开。否则未来会很难表达这些需求：

- 命中后销毁
- 命中后继续飞行
- 只命中单位，不命中特效
- 可穿透同类，不可穿透敌方
- 同一目标只命中一次
- 同一目标每隔一段时间可重复命中

### 3.3 出生即完整挂载，而不是出生后补逻辑

`missile` 的核心特征不是“它是一个特效”，而是“它是一个出生后立即参与运动与碰撞结算的投射物”。

所以推荐设计为：

- 创建 `missile` 实体
- 同一条 spawn 流程中挂上 `motion`
- 同一条 spawn 流程中挂上 `collider`
- 同一条 spawn 流程中挂上 `hit policy`
- 注册到系统

整个过程应该在逻辑上视为一次原子化的“发射”动作。

## 4. 推荐的核心模型

## 4.1 Entity：实体层

实体层只解决“对象是谁”。

推荐保留当前已有的三类实体概念：

- `unit`
- `effect`
- `missile`

但要弱化“通过继承表达行为”的倾向，增强“通过能力挂载表达行为”的倾向。

建议实体层只提供这些稳定能力：

- 生命周期：创建、删除、有效性判断
- 基础空间信息：位置、朝向、高度
- 身份信息：类型、阵营、来源者、目标者
- 运行时能力引用：`motion`、`collider`、`hit policy`

推荐认知如下：

- `unit`：长期存在，可选挂运动能力和碰撞能力
- `effect`：表现对象，可选挂运动能力，可选挂碰撞能力
- `missile`：投射物实体，默认出生就挂运动能力与碰撞能力

## 4.2 Transform Adapter：空间接口适配层

当前项目里，`unit` 和 `effect` 的位置设置接口不同。为避免运动器内部到处写类型分支，建议增加统一适配层。

目标是让运动系统只依赖这组抽象接口：

```lua
adapter.getPosition(actor) -> x, y, z
adapter.setPosition(actor, x, y, z)
adapter.setFacing(actor, x, y, z) -- 或 yaw/pitch
adapter.isAlive(actor) -> boolean
```

这样以后：

- `motion` 不必关心自己操作的是 `unit` 还是 `effect`
- `missile` 可以沿用 `effect` 的空间接口，但保有独立的碰撞语义

## 4.3 Motion：运动能力

`motion` 只负责“下一帧应该移动到哪里”。

它不应该直接负责：

- 枚举碰撞对象
- 判断碰撞响应
- 执行伤害或销毁

`motion` 内部建议只维护：

- `actor`
- `mode` 或 `solver`
- `velocity`
- `acceleration`
- `elapsed`
- `duration`
- `prev_position`
- `next_position`
- `is_paused`
- `on_end`

推荐结构示意：

```lua
motion = {
    actor = actor,
    solver = ChargeSolver,
    velocity = { x = 0, y = 0, z = 0 },
    acceleration = { x = 0, y = 0, z = 0 },
    elapsed = 0,
    duration = 1.5,
    prev_position = nil,
    next_position = nil,
    on_end = nil,
}
```

### 4.3.1 Motion Solver：轨迹求解器

你现在已有的 `Charge`、`Bezier3`、`Surround`、`Down` 很适合收敛成一组独立求解器。

每个求解器只做一件事：

- 输入当前状态
- 计算下一帧位姿
- 告诉系统是否自然结束

示意接口：

```lua
solver.step(motion, dt) -> result

result = {
    x = ...,
    y = ...,
    z = ...,
    facing = ...,
    ended = false,
}
```

这样每种运动方式都可替换，但主循环不变。

## 4.4 Collider：碰撞能力

`collider` 只负责回答一个问题：

“当前 actor 在这一帧会不会与某些对象发生碰撞？”

它不负责施加伤害，也不负责决定销毁。

### 4.4.1 Collider 应保存的状态

推荐字段：

- `owner`：所属实体
- `shape`：碰撞体形状，初期可以先只支持圆形
- `radius`
- `mask`
- `query_mode`
- `once_per_target`
- `hit_cooldown`
- `hit_record`

推荐示意：

```lua
collider = {
    owner = actor,
    shape = "circle",
    radius = 64,
    mask = {
        unit = true,
        missile = true,
        effect = false,
    },
    query_mode = "sweep",
    once_per_target = true,
    hit_cooldown = 0.0,
    hit_record = {},
}
```

### 4.4.2 为什么建议支持 sweep，而不只是 point check

如果每帧只检测“当前位置圆形碰撞”，高速 `missile` 容易穿模。

更稳妥的做法是：

- 记录上一帧位置 `prev_position`
- 记录这一帧候选位置 `next_position`
- 用这段路径做 sweep 检测

哪怕初期实现仍然是“分段采样”而不是真正数学意义的连续碰撞，也比纯当前位置检测更可靠。

## 4.5 Hit Policy：命中响应策略

`hit policy` 专门处理“碰撞已发生，接下来怎么办”。

这层可以非常灵活，因为它只依赖碰撞结果，不依赖轨迹细节。

推荐接口：

```lua
hit_policy.resolve(context) -> action
```

`context` 可以包含：

- `actor`
- `target`
- `motion`
- `collider`
- `hit_type`
- `hit_position`
- `time`

`action` 可以包含：

- `stop_motion`
- `destroy_actor`
- `continue_motion`
- `remove_collider`
- `damage`
- `emit_event`

### 4.5.1 常见策略类型

- `destroy_on_first_hit`
- `penetrate_n_targets`
- `periodic_hit`
- `bounce_on_hit`
- `ignore_friendly`
- `stop_on_blocker_only`

这样以后你写新技能时，重点就变成“组合已有能力”，而不是在 `Entry.lua` 里继续堆条件。

## 4.6 Spawn Spec：生成规格

为了保证 `missile` 出生即完整挂载，推荐引入“规格化创建”概念。

推荐把“发射一个导弹”视为一条高层指令，而不是几段零碎调用。

示意：

```lua
MissileService.spawn({
    actor = missile_entity,
    source = caster,
    target = target,
    motion = MotionSpec.charge({
        speed = 900,
        acceleration = 0,
    }),
    collider = ColliderSpec.circle({
        radius = 48,
        mask = { unit = true, missile = true },
        once_per_target = true,
    }),
    hit = HitSpec.destroy_on_first_hit({
        on_hit_unit = function(ctx) end,
        on_hit_missile = function(ctx) end,
    }),
})
```

这层的价值非常大：

- 统一创建路径
- 避免出现“创建了 missile，但忘了挂碰撞器”
- 让技能脚本更像声明配置，而不是手工组装过程

## 5. 推荐的系统拆分

## 5.1 MotionSystem

职责：

- 维护所有 active motion
- 每帧推进 motion
- 生成 `prev_position` 和 `next_position`
- 把位姿变更提交给空间适配层
- 处理自然结束

不负责：

- 枚举命中目标
- 处理命中效果

### 5.1.1 推荐帧流程

```text
for motion in active_motions:
    1. 读取 actor 当前坐标
    2. 记录 prev_position
    3. 调用 solver.step(motion, dt)
    4. 记录 next_position
    5. 临时或正式更新 actor 空间位置
    6. 通知 CollisionSystem 做检测
    7. 根据碰撞返回结果决定是否结束 motion
    8. 如自然结束，执行 on_end
```

## 5.2 CollisionSystem

职责：

- 维护所有 active collider
- 根据 `prev_position -> next_position` 检测碰撞
- 根据 `mask` 筛选目标类型
- 根据 `hit_record` 做去重或冷却
- 构造 `hit context`
- 调用 `hit policy`

不负责：

- 自己生成轨迹
- 自己决定运动方式

### 5.2.1 推荐的查询分层

第一层按目标大类分开：

- `unit query`
- `missile query`
- `effect query`

第二层按碰撞形状判断。

初期完全可以先做：

- `circle vs unit`
- `circle vs missile`

后续再扩展：

- capsule/sweep
- line
- sector
- box

## 5.3 LifecycleSystem

职责：

- 统一处理对象结束和清理
- 负责注销 `motion`
- 负责注销 `collider`
- 负责销毁 `missile/effect`
- 处理“命中后结束”和“自然结束”的收口逻辑

这一层存在的价值是避免：

- `Motion.cancel`
- `Effect.destroy`
- `Missile.destroy`
- `Timer.cancel`

这些行为散落在多个模块里，导致收尾链路混乱。

## 6. 一帧内的推荐执行顺序

这是整个框架里最重要的部分之一。

推荐顺序如下：

```text
1. Spawn 阶段
   创建新实体
   挂 motion / collider / hit policy
   注册到系统

2. Motion 阶段
   计算每个 actor 的下一帧位置

3. Collision 阶段
   基于 prev_position -> next_position 做碰撞检测

4. Resolve 阶段
   执行 hit policy
   生成 stop / destroy / continue 等动作

5. Commit/Cleanup 阶段
   统一执行销毁、注销、结束回调
```

建议避免在碰撞回调里直接马上深度删除一切对象。更稳妥的做法是：

- 先记录决议
- 帧末统一 commit

这样能避免遍历过程中删对象带来的状态不一致。

## 7. 三类实体的推荐职责

## 7.1 Unit

`unit` 是世界中的主实体。

推荐特性：

- 平时可常驻碰撞体
- 可被别人的碰撞器命中
- 自己在某些技能期间也可挂运动器

典型例子：

- 冲锋
- 击退
- 吸附
- 抛物线击飞

因此 `unit` 不应与“不可移动”绑定，也不应与“只有原生碰撞”绑定。

## 7.2 Effect

`effect` 更偏视觉表现层。

推荐特性：

- 默认没有碰撞语义
- 可挂运动器
- 在部分技能里可临时挂碰撞器

典型例子：

- 飞行刀光
- 旋转法阵
- 跟随弹道的视觉特效

如果一个对象只有表现，没有命中语义，那么它只需要 `motion`，不需要 `collider`。

## 7.3 Missile

`missile` 应视为“投射物实体”，而不是“会飞的特效别名”。

推荐特性：

- 默认出生即挂 `motion`
- 默认出生即挂 `collider`
- 默认出生即挂 `hit policy`
- 默认有自己的生命周期

也就是说，`missile` 不是比 `effect` 多一个 `hp` 或 `resist` 就够了，它的关键是运行时语义不同。

## 8. Missile 的完整推荐链路

这部分专门回答“为什么 missile 创建时就应该拥有 motion”。

推荐链路：

```text
技能或逻辑模块发出发射请求
-> MissileService.spawn(spec)
-> 创建 missile 实体
-> 绑定 source / target / team / owner
-> 构建 motion
-> 构建 collider
-> 构建 hit policy
-> 注册进 MotionSystem / CollisionSystem
-> 从下一帧开始参与更新
```

### 8.1 为什么不能先 create，再稍后补 motion

如果 `missile` 在创建到挂 motion 之间有空档，会有几个问题：

- 这一帧是否应该碰撞，语义不清晰
- 是否应该被其他 `missile` 命中，语义不清晰
- 调用链容易遗漏步骤
- 以后多人协作时容易出现“生成了对象但没有真正进入系统”的 bug

所以推荐把 `spawn missile` 视为统一入口。

### 8.2 你当前工程里的最好落点

结合现有工程，推荐入口优先级如下：

1. 新增 `Missile_Create_Event`
2. 新增 `Logic/Process/Missile/Create.lua`
3. 由 `MissileService.spawn` 统一完成能力挂载
4. `Missile.__init__` 只做实体初始化，不直接拼整套运行逻辑

如果前期不想立刻引入 `MissileService`，也至少应保证：

- 创建 `missile` 的地方不再手工分散做多个步骤
- 所有出生逻辑集中在一条统一路径上

## 10. 对当前 `Entry.lua` 字段的推荐拆分

当前 `Motion` 中的很多字段其实不该都属于 `motion`。

### 10.1 应继续留在 motion 的字段

- `actor`
- `origin`
- `target`
- `dur`
- `high`
- `angle`
- `is_pause`
- `surround`
- `angle_spd`
- `surround_radius`
- `speed`
- `speed_vector3`
- `acceleration`
- `acceleration_vector3`
- `charge_distance`
- `charge_distance_now`
- `start_point`
- `end_point`
- `control_one`
- `control_two`
- `control_t`
- `on_end`

这些字段大体属于轨迹和运动状态。

### 10.2 更适合拆到 collider 的字段

- `range`
- `knock_gap`
- `knock_cool_down`
- `is_collide_unit`
- `is_collide_missile`
- `reknock_unit_group`

这些字段本质上都属于碰撞检测和命中去重。

### 10.3 更适合拆到 hit policy 的字段

- `on_collision_with_unit`
- `on_collision_with_missile`

这些是典型的命中响应逻辑。

## 11. 推荐的数据流

推荐每个 `missile` 在运行时维持如下数据流：

```text
missile entity
  -> transform adapter
  -> motion
  -> collider
  -> hit policy
```

系统间的数据流：

```text
MotionSystem
  -> 产生位移结果
  -> 交给 CollisionSystem

CollisionSystem
  -> 产生命中上下文
  -> 交给 HitPolicy

HitPolicy
  -> 生成行动决议
  -> 交给 LifecycleSystem
```

这样以后调试会很清楚：

- 飞得不对，看 `motion`
- 判定不对，看 `collider`
- 命中效果不对，看 `hit policy`
- 死亡和收尾不对，看 `lifecycle`

## 12. 推荐的状态机

对 `missile` 来说，推荐存在明确的生命周期状态。

```text
created
-> spawned
-> active
-> hit_resolved
-> ending
-> destroyed
```

### 12.1 各状态含义

- `created`：实体刚构造，还未注册到系统
- `spawned`：能力已挂载完成，等待进入更新
- `active`：每帧正常运动和碰撞
- `hit_resolved`：本帧发生命中，已得出处理结果
- `ending`：等待统一清理
- `destroyed`：从系统注销且实体销毁

有了这个状态机之后，很多边界条件会更好处理。

## 13. 推荐的事件体系

你当前项目已有 `Unit_Create_Event` 和 `Unit_Delete_Event` 风格，说明事件体系已经成型。

推荐补充：

- `Missile_Create_Event`
- `Missile_Delete_Event`
- `Motion_Start_Event`
- `Motion_End_Event`
- `Collision_Event`
- `Missile_Hit_Event`

### 13.1 事件的作用

- 让创建和运行逻辑与技能逻辑解耦
- 让调试更方便
- 让以后扩展日志、表现、音效、统计更自然

但要注意：

事件适合表达“通知”，不适合替代系统主流程。

也就是说：

- 系统推进仍然要由 `MotionSystem / CollisionSystem` 掌控
- 事件只是向外广播结果

## 14. 推荐的配置层设计

后续如果技能数量变多，建议将创建参数做成“规格对象”。

### 14.1 MotionSpec

用于定义运动方式。

例子：

- `charge`
- `bezier3`
- `surround`
- `down`
- `homing`

### 14.2 ColliderSpec

用于定义碰撞方式。

例子：

- `circle`
- `sweep_circle`
- `line`
- `sector`

### 14.3 HitSpec

用于定义命中响应。

例子：

- `destroy_on_first_hit`
- `penetrate_3_targets`
- `tick_damage_aura`

配置分层后，技能书写会更接近“声明能力组合”。

## 15. 推荐的迁移顺序

这部分很重要，因为你不需要一次性重写全部逻辑。

### 阶段 1：先做职责认知切分

目标：

- 明确 `motion` 只算位移
- 明确 `collider` 只做检测
- 明确 `hit policy` 只做响应

这一步先统一团队心智模型，不急着全面改代码。

### 阶段 2：把现有 `Charge / Bezier3 / Surround / Down` 收敛成求解器

目标：

- 把不同运动方式独立成 solver
- 主循环只负责调用统一接口

收益：

- 以后新增轨迹不必改系统结构

### 阶段 3：抽出 CollisionSystem

目标：

- 把 `range`、`knock_gap`、`reknock_unit_group` 之类字段从 `motion` 语义中移出
- 先支持 `unit` 和 `missile` 两类碰撞即可

收益：

- `missile` 的命中语义从运动逻辑中解耦

### 阶段 4：引入 HitPolicy

目标：

- 将 `on_collision_with_unit`、`on_collision_with_missile` 抽成命中策略

收益：

- 技能差异不再污染底层系统

### 阶段 5：统一 `MissileService.spawn`

目标：

- 保证 `missile` 出生即完整挂载能力
- 统一创建路径

收益：

- 彻底消灭“先创建，后补挂”的链路不清晰问题

## 16. 这套设计下的典型示例

## 16.1 普通飞弹

特征：

- 匀速直线
- 命中第一个敌方单位后销毁

能力组合：

- `motion = charge`
- `collider = sweep_circle(unit = true)`
- `hit = destroy_on_first_hit`

## 16.2 可穿透飞弹

特征：

- 连续穿过多个目标
- 每个目标只命中一次

能力组合：

- `motion = charge`
- `collider = sweep_circle(once_per_target = true)`
- `hit = penetrate_n_targets`

## 16.3 旋转刀阵

特征：

- 围绕目标旋转
- 周期性命中附近单位

能力组合：

- `motion = surround`
- `collider = circle(unit = true, hit_cooldown = 0.2)`
- `hit = periodic_hit`

## 16.4 会撞飞弹的飞弹

特征：

- 同时检测 `unit` 与 `missile`
- 打到单位和打到飞弹时响应不同

能力组合：

- `motion = charge`
- `collider.mask = { unit = true, missile = true }`
- `hit.on_hit_unit = ...`
- `hit.on_hit_missile = ...`

这正是你当前需求最核心的场景。

## 17. 对“现代化设计”的具体理解

这里的“现代化”不等于一定要做 ECS，也不等于要把所有对象完全数据驱动。

更适合当前项目的现代化设计是：

- 职责清晰
- 系统与策略分离
- 创建链路原子化
- 扩展点前置
- 配置驱动能力组合
- 生命周期和清理逻辑统一

对于这个工程体量，推荐走“轻量组件化 + 系统驱动 + 事件辅助”的路线，而不是激进重写为完整 ECS。

原因是：

- 你已有明确的实体类
- 已有事件系统
- 已有按目录拆分的逻辑模块
- 现阶段最需要的是职责切分，不是范式革命

## 18. 最终建议

如果只保留一句最核心的设计建议，就是：

`missile` 应被设计为一种“出生时就完整挂载 motion + collider + hit policy 的投射物实体”，而不是“先创建出来、之后再逐步补行为的特效对象”。

对应到框架层面，推荐总链路为：

```text
实体创建
-> 能力挂载
-> 系统注册
-> 每帧运动
-> 每帧碰撞检测
-> 命中响应
-> 生命周期清理
```

其中：

- `motion` 只负责运动
- `collider` 只负责检测
- `hit policy` 只负责响应
- `service/process` 负责把创建链路串起来

这会让后续框架逻辑更清晰，也更适合在技能数量变多后继续演进。

## 19. 后续落地建议

如果后续要进入代码实现，建议按以下顺序开始：

1. 先整理 `Motion` 主入口，让它只保留“轨迹推进”职责。
2. 再单独抽 `CollisionSystem`，只先支持圆形碰撞和 `unit/missile` 两类对象。
3. 再引入 `HitPolicy`，把命中回调从运动系统中拆出来。
4. 最后统一 `MissileService.spawn`，把发射链路收敛成一个入口。

这样风险最小，也最符合当前工程的演进节奏。

## 20. 大型成熟项目中的通用架构

这一节刻意抛开当前项目，只讨论大型成熟项目通常如何组织 `motion`、`collision`、`damage`、`death`、`destroy` 这一整条链路。

结论先说在前面：

大型成熟项目里，通常不会让一个“运动器库”直接包办整条链路，也不会让“碰撞检测”直接到处改世界状态。更常见的做法是把运行流程拆成若干层，每层只做有限职责，并由统一的模拟调度层按固定顺序推进。

### 20.1 常见的五层拆法

在很多成熟项目里，运行时通常会被拆成下面几层：

```text
Intent / Spawn
-> Movement / Motion
-> Collision / Physics Query
-> Gameplay Resolve
-> Lifecycle / World Commit
```

各层的典型职责如下：

- `Intent / Spawn`
  - 接收技能、AI、输入、脚本发出的请求
  - 创建投射物、角色动作、范围效果
  - 生成本帧要进入世界的新对象和命令

- `Movement / Motion`
  - 只负责计算位移和姿态
  - 产出 `prev_transform -> next_transform`
  - 不直接处理伤害和死亡

- `Collision / Physics Query`
  - 负责碰撞体同步、空间查询、Broadphase、Narrowphase
  - 产出 contact、overlap、sweep hit 等结果
  - 不直接拍板“谁死谁活”

- `Gameplay Resolve`
  - 把碰撞结果翻译成游戏层结果
  - 比如伤害、护盾、格挡、反弹、穿透、吸血、触发器
  - 产出 action、event、state change request

- `Lifecycle / World Commit`
  - 统一把结果提交回世界状态
  - 例如扣血、判死、销毁对象、注销组件、回收资源

这套拆法的价值在于：

- 调试时可以快速定位问题属于哪一层
- 扩展新规则时不会污染整条链路
- 多人协作时职责边界清晰
- 网络同步、回放、重演、确定性模拟更好做

### 20.2 成熟项目中常见的三种实现路线

严格来说，大项目并不是只有一种写法，通常会在以下三种路线之间选择。

#### 20.2.1 纯 Gameplay 驱动

特点：

- 运动由游戏逻辑自己算
- 碰撞多用 query、sweep、overlap，而不是完整刚体模拟
- 投射物、技能、范围效果经常走这条链

优点：

- 可控性高
- 容易写穿透、锁定、贴体飞行、曲线弹道
- 更适合 RPG、MOBA、ACT、ARPG 技能系统

缺点：

- 需要自己维护碰撞状态和判定流程
- 对高速物体、复杂动态场景需要更多工程工作

#### 20.2.2 纯 Physics 驱动

特点：

- 交给刚体系统推进位置
- 碰撞、接触、反弹由物理引擎处理
- 游戏逻辑主要响应物理事件

优点：

- 低层碰撞和物理行为成熟
- 对真实物理互动场景更省力

缺点：

- 游戏规则可控性较差
- 很难优雅表达大量技能化、脚本化、非物理弹道
- 在强规则玩法里容易出现“物理正确但玩法不对”

#### 20.2.3 混合式架构

这是大型成熟项目中最常见的一类。

特点：

- 运动由游戏逻辑或角色移动系统控制
- 碰撞查询借助独立 Physics Scene 或 Query Scene
- 伤害、死亡、状态变化仍由 Gameplay 系统处理

也就是说：

- 位置推进不一定由物理引擎主导
- 但碰撞查询通常会借助统一的空间索引和查询系统
- 真正的伤害和销毁，一般仍归游戏逻辑层

这类设计通常最平衡：

- 保留玩法可控性
- 复用成熟碰撞查询能力
- 兼顾性能、调试性、扩展性

### 20.3 为什么成熟项目通常不让 Motion 直接处理 Collision

从工程规模上看，成熟项目通常会刻意避免 `motion` 和 `collision` 紧耦合，原因主要有以下几点。

#### 20.3.1 运动方式很多，但碰撞基础设施应复用

一个项目里可能同时有：

- 直线飞弹
- 曲线飞弹
- 跟踪飞弹
- 冲锋
- 击退
- 旋转刀阵
- 抛物线位移

这些对象的运动方式不同，但最终都要复用同一套空间查询和碰撞基础设施。

如果每个 `motion` 都自己处理碰撞：

- 重复代码会很多
- 命中过滤规则难以统一
- 性能优化很难集中做

#### 20.3.2 空间查询通常要集中管理

成熟项目里，碰撞往往不是“拿当前位置随手查一下”这么简单，而是依赖统一的空间结构：

- 网格
- 四叉树
- BVH
- 动态 AABB Tree
- Physics Scene

这些结构通常是全局或分区维护的，不适合散落在每个 `motion` 模块内部。

#### 20.3.3 碰撞之后往往有很长的业务链

一次命中不只是“撞到了”。

后面可能还要继续走：

- 命中判定
- 阵营过滤
- 护盾吸收
- 抗性减伤
- 伤害结算
- 击退/硬直
- 死亡判定
- 触发事件
- 特效/音效/飘字
- 掉落和经验

如果 `motion` 直接掌握整条链路，模块会迅速失控。

#### 20.3.4 立即删除对象会破坏模拟稳定性

成熟项目普遍会避免在“正在遍历碰撞结果”时直接删对象。

更常见的做法是：

- 本阶段只记录命中结果
- 生成 action / command / event
- 在提交阶段统一执行销毁和清理

这就是很多项目里都会出现 `command buffer`、`deferred queue`、`pending destroy list` 这类机制的原因。

#### 20.3.5 网络和回放要求固定顺序

大型项目常常需要：

- 服务器权威模拟
- 客户端预测
- 战斗回放
- 确定性重演

这些都要求本帧的处理顺序稳定可控。

因此更常见的是：

- `Movement` 先产出位移结果
- `Collision` 再统一生成命中
- `Resolve` 再统一处理业务后果
- `Commit` 最后统一写回世界

而不是多个模块互相嵌套调用、临时直接改状态。

### 20.4 成熟项目常见的系统职责分层

大型项目里，常见的分工通常接近下面这样。

#### 20.4.1 Movement System

职责：

- 维护速度、加速度、轨迹参数
- 计算当前帧位姿
- 更新 transform

不负责：

- 直接造成伤害
- 直接销毁对象

#### 20.4.2 Collision / Query System

职责：

- 管理碰撞体
- 同步碰撞体到查询世界
- 做 overlap、raycast、sweep、shape cast
- 产出 hit/contact

不负责：

- 最终决定业务结果

#### 20.4.3 Gameplay Hit / Ability Resolve

职责：

- 把碰撞结果解释成玩法结果
- 处理阵营、标签、豁免、穿透、反弹、触发条件
- 生成伤害请求和状态变化请求

这一层在大型项目中通常是最复杂的，因为它承载的是玩法规则，而不是几何判定。

#### 20.4.4 Damage / Attribute System

职责：

- 处理伤害公式
- 读取攻击、护甲、抗性、护盾、减伤
- 改写 HP、MP、状态

在很多成熟项目里，`damage` 本身就是独立系统，而不是碰撞系统的一部分。

#### 20.4.5 Death / Lifecycle / Destroy System

职责：

- 检查 HP 是否归零
- 状态从 `alive -> dying -> dead -> destroy_pending`
- 注销 movement、collision、AI、buff
- 统一执行资源清理和对象回收

这一层通常非常强调统一入口，避免任何系统私自深度销毁对象。

### 20.5 成熟项目中的一帧推荐顺序

一个较为典型的顺序如下：

```text
1. Gather Commands
   收集输入、AI、技能、生成请求

2. Spawn / Despawn Queue Apply
   把本帧新建对象注册进世界

3. Movement Update
   推进角色、飞弹、跟随物、轨迹物

4. Sync Collision Proxies
   把最新 transform 同步到碰撞查询世界

5. Broadphase / Narrowphase
   生成 overlap、contact、sweep hit

6. Gameplay Resolve
   生成命中、伤害、状态变化请求

7. Damage / Attribute Apply
   改写属性

8. Death / Lifecycle Commit
   判死、清理、销毁

9. Event / FX / Replication
   广播事件、表现、网络同步
```

这条顺序的关键不是名字，而是稳定性：

- 先移动
- 再碰撞
- 再结算
- 最后提交世界状态

### 20.6 物理碰撞与玩法命中的分离

成熟项目里，一个很重要但常被忽略的点是：

“物理接触”不等于“玩法命中”。

例如：

- 刀光特效穿过角色模型，但只在帧判定窗口里生效
- 子弹碰到了盾牌碰撞体，但玩法上应被格挡而不是打到角色本体
- 飞弹穿过友军单位的碰撞体，但玩法规则规定不命中友军
- 范围技能命中了多个目标，但其中一部分因为无敌状态不吃伤害

所以成熟项目里往往会分成两层：

- `Physics Contact / Collision Query`
- `Gameplay Hit Confirmation`

前者回答“几何上接触到了什么”，后者回答“玩法上是否算命中”。

### 20.7 为什么 Damage 往往独立于 Collision

在大型成熟项目里，`damage` 很少直接长在碰撞模块里，因为伤害来源不只是碰撞。

常见伤害来源还有：

- 持续伤害
- 光环
- 反伤
- 脚本事件
- 陷阱
- 自残
- 环境伤害
- 执行效果

如果伤害系统绑死在碰撞里，就无法统一公式、日志、护盾、减伤、吸血、战报和统计。

因此更常见的链路是：

```text
Collision / Spell / Aura / Script
-> DamageRequest
-> DamageSystem.apply()
-> Health changed
-> DeathSystem.check()
```

### 20.8 为什么 Destroy 往往独立于 Damage

同样，成熟项目通常也不会让 `damage` 模块直接到处销毁对象。

因为对象死亡之后，后面还可能涉及：

- 死亡动画
- 霸体/锁血/复活
- 死亡事件
- 尸体保留
- 延迟移除
- 掉落结算
- 经验结算
- 仇恨清理

所以更常见的是：

```text
DamageSystem
-> target.hp = target.hp - value
-> mark target as maybe_dead

DeathSystem / LifecycleSystem
-> confirm death rule
-> transition state
-> schedule destroy
```

### 20.9 大型项目里的依赖关系通常长什么样

如果用“依赖”和“调度”分开看，成熟项目里常见关系如下：

```text
MotionSolver
  不依赖 Collision

MovementSystem
  依赖 MotionSolver
  不直接依赖 Damage / Destroy

CollisionSystem
  读取 Transform / Motion 结果
  不直接拥有 Damage / Destroy

GameplayResolveSystem
  读取 Collision 输出
  生成 DamageRequest / StateChangeRequest

DamageSystem
  处理属性变化

LifecycleSystem
  处理死亡、销毁、清理

SimulationPipeline
  统一调度以上所有系统
```

这意味着：

- 数据上，`Collision` 会读取 `Motion` 的产物
- 工程上，`Motion` 不一定需要依赖 `Collision`
- 真正知道“先运动再碰撞再结算”的，通常是上层调度器

### 20.10 对本设计最有价值的行业结论

从成熟项目的通用做法提炼到这里，可以得到几条最值得保留的结论：

1. `Motion` 最适合只做位移推进，不适合同时承担碰撞、伤害、死亡、销毁。
2. `Collision` 最适合只做接触发现和命中候选生成，不适合直接深度改写世界。
3. `Damage` 最好独立出来，因为伤害来源远不止碰撞。
4. `Lifecycle` 最好独立出来，因为死亡和销毁涉及大量统一规则。
5. 真正把这几层串起来的，应该是固定顺序的 `SimulationPipeline` 或 `UpdatePipeline`。
6. 大型项目最常见的不是“某个模块拥有全部逻辑”，而是“每个系统只拥有一小段流程，整体由调度层串起来”。

## 21. 对当前文档的补充结论

结合上面的行业分析，可以把本文档前面的建议再补强成一句更完整的话：

推荐的设计不是“让 Motion 完全不知道碰撞存在”，而是“让 Motion 只负责产生可供碰撞系统消费的位移结果，并由统一调度层在运动之后驱动碰撞、伤害和生命周期提交”。

因此，更成熟的理解应该是：

```text
MotionSolver
-> 只算运动

CollisionSystem
-> 读取运动结果，生成命中

DamageSystem / GameplayResolve
-> 解释命中结果

LifecycleSystem
-> 统一提交死亡和销毁

UpdatePipeline
-> 统一调度整个顺序
```

这套结构比“一个 Motion 库内部顺手做完所有事情”更接近大型成熟项目的长期演进方式。
